## **1. 范围**

本标准规定了在噪声信道与受限带宽环境下进行高效流式文本传输的协议格式、物理层调制参数、链路控制机制及会话管理规范。

本标准适用于：
- 短波（HF）天波与地波数字通信；
- 海事 VHF/UHF 数据链路扩展；
- 低轨卫星物联网（LEO-IoT）文本回传；
- 应急与灾害通信网络的高吞吐文本广播；
- 任何需要从 v1.0 固定块架构升级至流式传输的 legacy 系统。

**核心改进目标**：针对 ESSO 10001-2026（v1.0）的固定小包停等瓶颈，引入变长帧、滑动窗口 ARQ、自适应物理层与级联前向纠错，将信道占用率从 <30% 提升至 >85%，并将典型文本传输延迟降低一个数量级。

## **3. 术语和定义**

**3.1 帧（Frame）**  
v2.0 的基本传输单元，采用 HDLC 类定界符，Payload 长度可变（1 ~ 512 字节）。

**3.2 虚电路（Virtual Circuit）**  
传输层建立的逻辑通信通道，具有状态（连接/数据传输/断开）和配置参数。

**3.3 自适应混合调制（AHM, Adaptive Hybrid Modulation）**  
根据实时信噪比（SNR）在无握手情况下动态切换调制阶数的物理层机制。

**3.4 级联纠错（Concatenated FEC）**  
外码（Reed-Solomon）与内码（LDPC-Staircase）串联使用的纠错架构。

**3.5 选择性重传（SR-ARQ, Selective Repeat ARQ）**  
滑动窗口机制下仅重传丢失或损坏帧的链路控制协议。

**3.6 导频模式识别（Pilot Pattern Recognition）**  
接收端通过前导中的 Mode Indicator 自动识别当前调制模式，无需显式信令协商。

## **6. 物理层 (PHY) 增强规范**

### **6.1 自适应混合调制 (AHM)**

根据实时 SNR 动态切换调制模式。**无需握手协商**，接收端通过导频结构中的 Mode Indicator 自动识别。

| 模式 | 调制 | 符号率 | 比特率 | 最小 SNR | 典型场景 |
|:----:|:-----|:------:|:------:|:--------:|:---------|
| **N0** | MFSK-16 | 100 Bd | 400 bps | 6 dB | 极弱信号 / 短波天波 / v1.0 兼容 |
| **N1** | MFSK-32 | 200 Bd | 1000 bps | 10 dB | 城市 VHF/UHF / 噪声中等 |
| **N2** | MFSK-64 | 300 Bd | 1800 bps | 14 dB | 直射视距 / 海事移动 |
| **N3** | OFDM-64 | 600 Bd | 2400 bps | 18 dB | 高速数据 / 海事卫星 / 低轨卫星 |

**频率规划**（保留 1000 ~ 3000 Hz 音频带宽兼容性）：
- **MFSK 模式**：沿用 v1.0 频率栅格，按符号率比例扩展；
- **OFDM 模式**：64 子载波，子载波间隔 31.25 Hz，有效带宽 2000 Hz；
- **保护带**：上下各 500 Hz 滚降，防止声码器与信道滤波器失真。

### **6.2 导频结构**

每帧前导（Preamble）格式：

```
[AGC Burst] [Sync Word] [Mode Indicator]
   16 符号    2 符号       4 符号
```

- **AGC Burst**：固定序列 `0xAA` 重复 16 次（交替高低频符号），用于接收端 AGC 锁定与频偏粗估计；
- **Sync Word**：固定 `0x7E7E`，帧定界与符号同步；
- **Mode Indicator**：当前模式编码（N0=00, N1=01, N2=10, N3=11），冗余发送 4 次，接收端采用多数表决（Majority Vote）解码，确保模式识别可靠性 ≥ 99.9%。

### **6.3 多径抵抗**

- **循环前缀 (CP)**：OFDM 模式（N3）添加 1/8 符号时长的循环前缀，可抵抗 20 ms 以内多径时延扩展；
- **频域均衡**：MFSK 模式采用 Goertzel 算法配合频域一阶均衡，补偿频率选择性衰落；
- **自动频率控制 (AFC)**：通过 AGC Burst 估计载波频偏，动态调整本地振荡器或数字下变频参数，容忍 ±50 Hz 频偏。

## **7. 链路层 (L2) — 流式变长帧**

### **7.1 帧格式**

废除 v1.0 固定 64B 块，采用 HDLC 类定界符 + 变长 Payload：

```
[Flag] [Header] [Payload] [FCS] [Flag]
 0x7E  3-6 B   1-512 B   4 B   0x7E
```

**定界规则**：
- 帧起始与结束标志均为 `0x7E`；
- Payload 中若出现 `0x7E`，采用字节填充（Byte Stuffing）：发送 `0x7D 0x5E`；
- Payload 中若出现 `0x7D`，发送 `0x7D 0x5D`。

### **7.2 Header 结构**

| 位域 | 长度 | 说明 |
|------|------|------|
| `VERSION` | 2 bit | 协议版本 (`10` = v2.0) |
| `TYPE` | 3 bit | `000`=数据, `001`=ACK, `010`=NACK, `011`=控制, `100`=心跳 |
| `MODULATION` | 2 bit | 当前物理层模式 (N0-N3) |
| `RESERVED` | 1 bit | 保留，置 0 |
| `SEQ` | 16 bit | 序列号 (0 ~ 65535，循环使用) |
| `ACK_SEQ` | 16 bit | 确认序号（仅 ACK/NACK 帧携带）|
| `LENGTH` | 9 bit | Payload 长度 (0 ~ 511) |
| `FLAGS` | 7 bit | 压缩/加密/分片/终止/紧急标志 |

**头开销**：
- 数据帧：3 字节（VERSION ~ LENGTH 的低 1 bit）
- ACK/NACK 帧：6 字节（含 ACK_SEQ）

相比 v1.0 的 16 字节固定头部，**效率提升约 80%**。

### **7.3 FLAGS 位定义**

```
Bit 6: 紧急标志 [URGENT]   0 = 普通, 1 = 紧急（可抢占）
Bit 5: 压缩标志 [COMP]     0 = 未压缩（仅控制帧）, 1 = LZ4 压缩
Bit 4: 加密标志 [CRYPT]    0 = 明文, 1 = AES-256-GCM 加密
Bit 3: 分片标志 [FRAG]     0 = 完整, 1 = 分片（见 7.5）
Bit 2: 终止标志 [FIN]      0 = 继续, 1 = 会话终止
Bit 1: 重传标志 [RETX]     0 = 原始, 1 = 重传帧
Bit 0: 保留 [RESV]         置 0
```

### **7.4 选择性重传 ARQ (SR-ARQ)**

**滑动窗口机制**：
- **窗口大小 W**：固定 8 帧（平衡内存占用与信道吞吐）；
- **流水线传输**：发送端无需等待确认，连续发送最多 8 帧；
- **批量确认**：ACK 帧携带：
  - `ACK_SEQ`：最后连续正确帧的序列号；
  - `Bitmap`：8 bit，指示窗口内各帧接收状态（bit i = 1 表示 SEQ = ACK_SEQ + 1 + i 已正确接收）。

**示例**：接收窗口内帧 0,1,2,4,5（丢失帧 3），ACK 回复 `ACK_SEQ=2, Bitmap=0b00011000`（bit3 置 1 表示帧 3 缺失）。

**超时与退避**：
- **基本超时**：`T_base = 2 × RTT_max`，默认 5 秒（适应短波长传播延迟）；
- **重传退避**：指数退避，最大重传次数 4 次，之后丢弃并上报传输层 `ERR_LINK_TIMEOUT`；
- **快速重传**：收到 3 次重复 ACK（相同 ACK_SEQ）立即重传对应帧，无需等待超时。

### **7.5 分片与重组**

当数据超过单帧最大 Payload（512B）或受 FEC 编码限制时启用分片：

| 标志组合 | 含义 |
|:--------:|:-----|
| `FRAG=1, FIN=0` | 中间分片 |
| `FRAG=1, FIN=1` | 最后分片 |
| `FRAG=0, FIN=1` | 单帧完整消息且终止 |
| `FRAG=0, FIN=0` | 单帧完整消息（继续）|

重组缓冲区超时 300 秒，超时后丢弃全部碎片。

### **7.6 级联纠错 (Concatenated FEC)**

**外码（Outer Code）**：Reed-Solomon (255, 223, t=16)
- 每 223 字节 Payload 计算 32 字节 RS 校验；
- 可纠正每编码块最多 16 字节错误（突发错误）。

**内码（Inner Code）**：LDPC-Staircase（符合 RFC 5170）
- 编码率自适应：0.5（高鲁棒）、0.75（平衡）、0.875（高速）；
- **软判决解码**：利用 MFSK 解调输出的符号置信度（Log-Likelihood Ratio），增益 2 ~ 3 dB。

**交织深度**：可配置 4 / 8 / 16 / 32，按信道质量动态调整：
- 深交织（32）：抗长突发噪声，延迟增加；
- 浅交织（4）：低延迟，适合交互式通信。

## **8. 传输层 (TL) — 会话与复用**

### **8.1 虚电路会话管理**

建立有状态的虚电路，解决 v1.0 无连接限制：

```
CONNECT  → 协商：压缩算法、窗口大小、目标呼号、加密参数
DATA     → 双向流式数据传输
DISC     → 优雅关闭，刷新缓冲区，发送 FIN 帧
```

**状态机**：

```
IDLE ──CONNECT──→ ESTABLISHED ──DATA──→ ESTABLISHED
                      │                  │
                      └────DISC────→ CLOSED
```

**心跳机制 (Keep-Alive)**：
- 空闲 30 秒发送 `TYPE=心跳` 帧（Payload 为空）；
- 90 秒未收到任何有效帧视为链路中断，触发重连或上报 `ERR_SESSION_LOST`。

### **8.2 多路复用**

单物理链路支持 **16 条逻辑信道（Channel ID 0 ~ 15）**：

| 信道 | 用途 |
|:----:|:-----|
| Ch 0 | 控制信道（会话管理、紧急广播、能力宣告）|
| Ch 1 ~ 14 | 数据信道（可分配给不同用户、应用或优先级流）|
| Ch 15 | 带外管理（远程配置、固件升级、诊断）|

**优先级抢占**：
- `FLAGS Bit 6 [URGENT] = 1` 标记紧急帧；
- 发送端可中断当前普通帧传输，立即插发紧急帧；
- 接收端按优先级缓冲重排序，确保紧急数据优先递交应用层。

## **9. 应用层 (AL) 优化**

### **9.1 强制压缩**

v2.0 所有文本数据**强制预压缩**，废除 v1.0 的可选压缩：

- **算法**：LZ4 帧格式（LZ4 Frame Format）；
- **预置字典**：4 KB 常用词汇字典（中英双语无线电术语、海事缩略语、应急关键词），压缩比额外提升约 20%；
- **零拷贝**：压缩后数据直接封装为 L2 Payload，避免中间缓冲区拷贝。

**典型性能**：
- 英文电报文本：压缩率 40%，等效吞吐提升 2.5 倍；
- 中文 UTF-8：压缩率 60%，等效吞吐提升 1.6 倍。

### **9.2 UTF-8 流式处理**

废除 v1.0 复杂的块内截断逻辑，采用**流式字节边界**：

- **发送端**：维护滑动文本字节流，按长度填充 Payload，无需关心 UTF-8 字符边界；
- **接收端**：使用流式 UTF-8 解码器（如 libiconv 流模式或自研状态机），自动处理跨帧字符拼接；
- **错误恢复**：遇到非法 UTF-8 序列时替换为 `U+FFFD`（�），不中断后续传输。

## **10. 互操作与向后兼容**

### **10.1 v1.0 兼容模式 (Legacy Mode)**

设备启动后必须监听 v1.0 同步字 `0xE7545445`：
- 若检测到 v1.0 信号，自动降级至 **N0 模式 + 固定 64B 块 + 停等 ARQ**；
- 兼容模式下所有 v2.0 增强特性关闭，行为与 ESSO 10001-2026 完全一致；
- 支持 v1.0/v2.0 网关转发（协议转换器功能）。

### **10.2 能力宣告信标 (Capability Beacon)**

每 60 秒在 Ch 0 广播一次 **Capability Announcement**：

```
[EUTTE/2.0] [Callsign: 8B] [Supported Modes Bitmap: 1B] [Current Channel Load: 1B]
```

- **Supported Modes Bitmap**：bit i = 1 表示支持 Ni 模式；
- **Current Channel Load**：当前各信道占用率百分比（0-100），辅助远端选择最佳通信参数与信道。

## **11. 安全与隐私要求**

### **11.1 明文与加密模式**

- **默认模式**：`FLAGS Bit 4 [CRYPT] = 0`，Payload 为 LZ4 压缩后的明文数据；
- **加密模式**：`FLAGS Bit 4 [CRYPT] = 1` 时：
  - 算法：**AES-256-GCM**（符合 FIPS 197 / NIST SP 800-38D）；
  - Nonce 派生：`Nonce = HMAC-SHA256(SEQ || CHANNEL_ID || Session Salt)[0:12]`；
  - 认证标签：占用 Payload 最后 16 字节；
  - 加密前须先压缩，即 **压缩 → 加密 → 封装**；
  - 实际可用数据：最大 Payload 缩减 16 字节。

### **11.2 源认证与完整性**

- **预共享密钥（PSK）**：会话建立时通过 CONNECT 协商或预配置，长度不少于 256 bit；
- **抗重放保护**：接收端维护已接收 `SEQ` 的滑动窗口（大小 1024），拒绝重复或乱序的过时帧；
- **伪造防护**：GCM 认证标签防止伪造 ACK/NACK 攻击。

## **13. 实现要求**

### **13.1 最小实现 (Minimal Implementation)**

任何声明符合 ESSO 10001.20 的设备必须支持：

1. **N0 模式**（MFSK-16 @ 100 Bd）用于 v1.0 兼容与极弱信号；
2. **N1 模式**（MFSK-32 @ 200 Bd）用于常规通信；
3. **滑动窗口 SR-ARQ**，窗口大小 W ≥ 4；
4. **LZ4 压缩/解压**（含帧格式支持）；
5. **HDLC 帧定界与字节填充**；
6. **v1.0 Legacy Mode 检测与自动降级**。

### **13.2 完整实现 (Full Implementation)**

完整实现额外支持：

- **N2/N3 模式**与自适应调制切换（AHM）；
- **LDPC-Staircase 内码**软判决解码；
- **16 信道多路复用**与优先级抢占；
- **AES-256-GCM 加密**（TLS-PSK 模式）；
- **RS(255,223) 外码**（可通过硬件加速）；
- **可配置交织深度**（4/8/16/32）。

## **附录 B（资料性）v1.0 至 v2.0 迁移指南**

### **B.1 软件升级路径**

1. **阶段一（双栈运行）**：设备同时监听 v1.0 同步字与 v2.0 帧标志，自动识别对端协议版本；
2. **阶段二（能力宣告）**：v2.0 设备每 60 秒广播 Capability Beacon，引导 v1.0 网关升级；
3. **阶段三（纯 v2.0）**：网络中所有节点确认支持 v2.0 后，关闭 Legacy Mode 以节省功耗。

### **B.2 硬件升级建议**

- **DSP 算力**：LDPC 软判决解码需额外 20 ~ 50 MIPS，建议采用 Cortex-M4 或更高；
- **内存需求**：SR-ARQ 窗口（8 帧 × 512 B）+ 交织缓冲区（32 × 512 B）约需 20 KB RAM；
- **频谱扩展**：N3 模式 OFDM 需支持 3000 Hz 音频带宽，传统 2700 Hz 滤波器需更换为 3300 Hz。
