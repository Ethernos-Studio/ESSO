## 1. 范围与目标

本规范定义 Ethernos 生态系统中软件完整性校验与身份验证所必须使用的密码学算法标准，包括：

- 哈希算法的选择与使用场景；
- 数字签名算法的参数与格式；
- 密钥生成、存储、轮换与退役的全生命周期管理；
- 签名与验证的标准化流程。

本规范为 ESSO-10430（包管理器证书规范）及 ESSO-11420（网络来源处理规范）提供密码学层面的实现依据。

## 3. 术语与定义

| 术语 | 定义 |
|------|------|
| **哈希（Hash）** | 将任意长度输入映射为固定长度输出的单向密码学函数，用于完整性校验。 |
| **数字签名（Digital Signature）** | 使用私钥对数据摘要进行加密运算，生成可验证来源与完整性的密码学证据。 |
| **Ed25519** | 基于 Edwards-curve 25519 的 EdDSA 签名方案，提供 128 位安全强度。 |
| **密钥对（Key Pair）** | 由公钥与私钥组成的非对称密码学密钥集合，公钥公开，私钥保密。 |
| **密钥指纹（Key Fingerprint）** | 公钥的密码学哈希值，用于快速标识与比对公钥。 |
| **签名载荷（Signing Payload）** | 被送入签名算法的规范化数据内容，通常为待签名对象的规范化 JSON 或二进制摘要。 |
| **双重签名（Dual Signature）** | 由发布者签名与官方认证机构签名共同组成的签名结构，提供两层独立验证。 |

## 4. 哈希算法标准

### 4.1 强制算法：SHA-256

**SHA-256**（Secure Hash Algorithm 256-bit）为 Ethernos 生态当前强制使用的唯一哈希算法。

- **标准**：FIPS 180-4
- **输出长度**：256 位（32 字节，64 字符十六进制表示）
- **使用场景**：
  - 软件分发包的完整性校验（`package_sha256`）；
  - 公钥指纹生成（取 SHA-256 前 128 位，即 16 字节/32 字符十六进制）；
  - 证书文件的内容寻址与缓存校验。

### 4.2 哈希表示规范

- 十六进制字符串必须为小写；
- 不允许使用 Base64 或其他编码表示哈希值；
- 在 JSON 与证书文件中，哈希字段命名必须为 `*_sha256`（如 `package_sha256`、`commit_sha256`）。

### 4.3 未来算法：SHA-3 / Keccak-256

若 SHA-256 出现密码学弱点或碰撞攻击可行化：
- 规范版本将升级至 2.0.0；
- SHA-3-256（Keccak-256）将作为替代算法引入；
- 过渡期内允许同时存在 SHA-256 与 SHA-3-256 字段，客户端优先校验 SHA-3-256。

## 5. 签名算法标准

### 5.1 强制算法：Ed25519

**Ed25519** 为 Ethernos 生态当前强制使用的唯一数字签名算法。

- **标准**：RFC 8032
- **曲线**：Edwards-curve 25519
- **安全强度**：128 位
- **私钥长度**：32 字节
- **公钥长度**：32 字节
- **签名长度**：64 字节

### 5.2 选择理由

- **性能**：Ed25519 的签名与验证速度显著优于 RSA 与 ECDSA；
- **安全性**：不受椭圆曲线随机数生成器故障影响，签名过程确定性；
- **简洁性**：公钥与签名长度固定，便于在网络传输与 JSON 中编码。

### 5.3 禁用算法

以下算法在 Ethernos 生态中被明确禁止用于新签发：
- RSA（所有密钥长度）
- ECDSA（secp256k1、secp256r1 等）
- DSA
- MD5、SHA-1（禁止用于任何安全相关哈希）

## 6. 密钥管理规范

### 6.1 密钥生成

- 必须使用密码学安全的随机数生成器（CSPRNG）生成私钥；
- 私钥生成应在发布者本地可信环境中完成，禁止通过在线服务生成；
- 推荐工具：OpenSSL 3.x、`libsodium`、标准库 `ed25519` 实现。

### 6.2 密钥存储

- **私钥**：必须存储于硬件安全模块（HSM）、安全密钥库或至少经过加密的本地文件系统；
- **公钥**：必须公开，并注册至 `cert.cavvy.ethernos.net` 对应的指纹元信息文件中。

### 6.3 密钥轮换

- 发布者应定期轮换签名密钥（推荐周期：24 个月）；
- 轮换时，新公钥必须提前至少 30 天注册至指纹元信息文件并标记为 `active`；
- 旧公钥在轮换后保留至少 12 个月，标记为 `expired` 或 `revoked`，用于验证历史版本；
- 若私钥泄露，必须立即标记为 `revoked`，并发布安全公告。

### 6.4 官方根密钥

- Ethernos 官方认证机构持有根密钥对，用于对官方安全源进行第二层签名；
- 根密钥私钥必须离线存储于 HSM，仅通过受控流程进行签名操作；
- 根密钥公钥硬编码于所有兼容客户端，或可通过独立可信通道获取。

## 7. 签名格式与流程

### 7.1 签名载荷构造

签名前，必须将待签名对象按以下步骤处理为签名载荷：

1. **提取字段**：移除签名对象中的 `signatures` 字段，保留其余所有字段；
2. **规范化序列化**：按 **RFC 8785（JSON Canonicalization Scheme, JCS）** 进行规范化 JSON 序列化；
   - 对象键按 Unicode 码点排序；
   - 无多余空白字符；
   - 数字使用最短精确表示；
3. **编码**：将规范化 JSON 字符串编码为 UTF-8 字节序列；
4. **摘要（可选）**：直接对 UTF-8 字节序列进行 Ed25519 签名，或先计算 SHA-256 摘要再签名。推荐直接对原始字节签名。

### 7.2 签名结构

签名在 JSON 中的表示：

```json
{
  "key_id": "a1b2c3d4e5f6a7b8",
  "algorithm": "Ed25519",
  "signature": "base64_encoded_signature"
}
```

- `key_id`：公钥指纹，取 SHA-256 前 16 字节（32 字符十六进制），用于快速匹配公钥；
- `algorithm`：固定为 `"Ed25519"`；
- `signature`：Base64（标准 RFC 4648）编码的 64 字节原始签名值。

### 7.3 双重签名流程

```
发布者签名
    │
    ▼
生成签名载荷
    │
    ▼
使用发布者私钥签名 → 生成 publisher.signature
    │
    ▼
提交至官方认证服务器
    │
    ▼
官方验证发布者签名有效
    │
    ▼
使用官方根密钥私钥签名 → 生成 authority.signature
    │
    ▼
合并为双重签名结构，写入 .cert 文件
```

## 8. 验证协议

### 8.1 验证步骤

客户端验证签名时，必须执行以下步骤：

1. **获取公钥**：从 `cert.cavvy.ethernos.net/{fingerprint}.json` 中提取对应 `key_id` 的公钥；
2. **重构载荷**：按 7.1 节规则重构签名载荷（移除 `signatures`，JCS 规范化）；
3. **验证发布者签名**：使用发布者公钥验证 `signatures.publisher.signature`；
4. **验证官方签名**：使用硬编码或预置的官方根公钥验证 `signatures.authority.signature`；
5. **结果判定**：两层签名均通过视为验证成功；任一层失败视为验证失败。

### 8.2 失败处理

| 失败场景 | 客户端行为 |
|----------|-----------|
| 公钥不存在或已撤销 | 拒绝安装，提示密钥无效。 |
| 签名值格式错误 | 拒绝安装，提示证书损坏。 |
| 签名验证失败 | 拒绝安装，提示内容可能被篡改。 |
| 算法字段非 Ed25519 | 拒绝安装，提示不支持的算法。 |
| 官方签名缺失 | 视为未通过官方认证，按未验证源处理。 |

## 9. 证书与公钥格式

### 9.1 公钥注册格式

在指纹元信息文件（`.json`）中注册的公钥：

```json
{
  "key_id": "a1b2c3d4e5f6a7b8",
  "algorithm": "Ed25519",
  "public_key": "MCowBQYDK2VwAyEA...",
  "activated_at": "2025-01-15T08:00:00Z",
  "status": "active"
}
```

- `public_key`：Base64 编码的 32 字节原始 Ed25519 公钥。

### 9.2 证书文件格式

参见 **ESSO-10430** 第 5.2 节。本规范仅约束其中密码学相关字段的算法与格式。

## 10. 算法迁移策略

### 10.1 迁移触发条件

以下任一条件触发算法迁移评估：
- 当前算法出现已知实用化攻击；
- 国家标准或行业规范强制要求升级；
- 社区广泛共识认为当前算法安全余量不足。

### 10.2 迁移流程

1. 发布新规范版本（如 2.0.0），定义新算法标准；
2. 官方服务器同时支持新旧算法至少 12 个月；
3. 客户端更新以支持新算法，优先使用新算法验证；
4. 过渡期结束后，旧算法签发的证书标记为过期；
5. 完全退役旧算法支持。
