## 1. 范围与适用性

本规范定义了 Ethernos 生态系统中，包管理器对**官方认证安全源**进行身份验证与完整性校验时所使用的数字证书格式、发放机制及服务器接口标准。

本规范适用于：
- **cavly** —— Cavvy 编程语言官方包管理器；
- 任何声明兼容 Ethernos ESSO-10430 的第三方包管理器；
- 官方认证安全源服务器（`caypak.ethernos.net`）与官方认证安全证书服务器（`caycert.ethernos.net`）的实现与维护。

---

## 2. 规范性引用

- **ESSO-10400** —— Ethernos 软件安全基础框架
- **ESSO-10410** —— Ethernos 数字签名与哈希标准
- **RFC 4122** —— UUID 标准
- **RFC 8785** —— JSON 规范化表示（JSON Canonicalization Scheme, JCS）
- **FIPS 180-4** —— SHA-256 安全哈希标准

## 3. 术语与定义

| 术语 | 定义 |
|------|------|
| **cavly** | Ethernos 官方 Cavvy 包管理器实现，兼容本规范。 |
| **包指纹（Package Fingerprint）** | 分配给每个认证包的唯一、持久、不可变更的标识符，采用 UUID v4 格式。该指纹与包的版本、发布历史、元数据变更无关，终身绑定于该包实体。 |
| **安全源（Secure Source）** | 经过 Ethernos 官方认证，列入 `caypak.ethernos.net` 索引的代码仓库或二进制分发源。 |
| **证书（Certificate）** | 由 `caycert.ethernos.net` 签发的 JSON 格式数字凭证，证明特定包指纹对应的包在特定版本下的完整性与来源可信性。 |
| **包指纹元信息（Fingerprint Metadata）** | 存储于 `caycert.ethernos.net/{fingerprint}.json` 中的包级静态信息，不随版本变化。 |
| **包指纹证书（Fingerprint Certificate）** | 存储于 `caycert.ethernos.net/{fingerprint}.cert` 中的版本级动态签名数据。 |

## 4. 证书体系架构

本规范采用**双层证书架构**：

```
┌─────────────────────────────────────────┐
│         官方认证安全源服务器              │
│    caypak.ethernos.net           │
│  ┌─────────────┐  ┌─────────────────┐ │
│  │  index.html   │  │   index.json      │ │
│  │  (GA生成)     │  │  (仓库静态文件)    │ │
│  └─────────────┘  └─────────────────┘ │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│         官方认证安全证书服务器            │
│    caycert.ethernos.net              │
│  ┌─────────────────────────────────────┐│
│  │  {fingerprint}.json  (元信息)       ││
│  │  {fingerprint}.cert  (证书)         ││
│  └─────────────────────────────────────┘│
└─────────────────────────────────────────┘
```

### 4.1 设计原则
- **持久性**：包指纹一经分配，终身不变，确保下游依赖锁定机制的稳定性。
- **分离性**：元信息（静态）与证书（动态版本级）分离存储，减少重复传输。
- **可验证性**：任何客户端均可仅通过官方服务器与本地哈希计算完成完整验证链，无需第三方 CA。
- **透明性**：所有索引与证书数据均托管于 GitHub Pages，变更历史可通过 Git 追溯。

## 5. 证书格式规范

- 发布者公钥轮换（需保留历史公钥列表）
- 仓库地址变更（需保留历史地址列表）

### 5.3 签名数据规范

签名对象（`signatures.*.signature`）的生成遵循以下步骤：

1. **构造签名载荷（Payload）**：
   取证书对象中除 `signatures` 字段外的所有内容，按 **RFC 8785 (JCS)** 进行规范化 JSON 序列化。

2. **载荷字符串**：
   序列化后的 UTF-8 字节序列。

3. **签名算法**：
   使用 Ed25519 对载荷进行签名。

4. **双重签名要求**：
   - **发布者签名**：证明发布者承认该版本内容。
   - **官方签名**：证明 Ethernos 官方已审核并纳入安全源索引。

## 6. 包指纹（Package Fingerprint）规范

### 6.1 格式要求
- 采用 **UUID Version 4** 标准格式（8-4-4-4-12 十六进制，含连字符）。
- 示例：`a1b2c3d4-e5f6-4a7b-8c9d-0e1f2a3b4c5d`

### 6.2 持久性保证
- 包指纹在首次通过官方认证时由 `caycert.ethernos.net` 生成并分配。
- **终身不变**：不因版本升级、名称变更、仓库迁移、发布者转移而改变。
- 若包被永久移除，其指纹进入 **退役（Retired）** 状态，保留于索引中但标记为 `retired`，不可重新分配给其他包。

### 6.3 生成时机
- 开发者首次向 `caypak.ethernos.net` 提交认证申请时，由官方 CI 流水线生成。
- 生成后写入 `index.json` 并同步创建 `caycert.ethernos.net/{fingerprint}.json` 骨架文件。

## 8. 证书发放与更新流程

### 8.1 首次认证流程

```
开发者提交申请
    │
    ▼
┌─────────────────┐
│  官方审核仓库    │  ← 人工 + 自动化扫描
│  (GitHub PR)     │
└─────────────────┘
    │
    ▼
生成 UUID 指纹 → 创建 {fingerprint}.json
    │
    ▼
开发者发布版本 → 提交 commit_hash & package_sha256
    │
    ▼
生成 {fingerprint}.cert（双重签名）
    │
    ▼
更新 index.json → 合并至主分支
    │
    ▼
GitHub Pages 自动部署
```

### 8.2 版本更新流程

1. 开发者发布新版本，提供：
   - `version`（SemVer）
   - `commit_hash`（Git 完整哈希）
   - `package_sha256`（分发包 SHA-256）
2. 官方 CI 验证：
   - 仓库地址与 `commit_hash` 是否可访问且一致；
   - `package_sha256` 是否与构建产物匹配；
   - 版本号是否遵循 SemVer 且高于历史版本。
3. 生成新的 `{fingerprint}.cert` 文件；
4. 更新 `index.json` 中该包的 `latest_version`、`latest_commit`、`latest_sha256`；
5. 提交至仓库，GitHub Pages 自动同步。

### 8.3 元信息变更流程

当发生以下变更时，需更新 `{fingerprint}.json` 而非 `.cert`：

- 包名称变更
- 发布者变更
- 仓库地址迁移
- 公钥轮换

**要求**：
- 新值写入 `current_*` 字段；
- 旧值追加至 `history` 对应数组；
- 必须附带 `reason` 说明变更原因；
- 变更后的 `.json` 需由官方重新签名（`esso_signature` 字段，可选扩展）。

## 9. 客户端验证协议

兼容 cavly 及本规范的包管理器，在安装或更新包时必须执行以下验证链：

### 9.1 验证步骤

```text
1. 从 caypak.ethernos.net/index.json 获取包索引
2. 提取目标包的 fingerprint、latest_sha256、cert_url
3. 下载 caycert.ethernos.net/{fingerprint}.cert
4. 验证证书内 name / publisher / repository 与 index.json 一致
5. 验证证书内 commit_hash 与仓库公开记录一致（可选深度验证）
6. 下载实际分发包
7. 本地计算 SHA-256，与证书内 package_sha256 比对
8. 验证证书双重签名（publisher + authority）
9. 验证通过，执行安装
```

### 9.2 失败处理

| 失败场景 | 客户端行为 |
|----------|-----------|
| `package_sha256` 不匹配 | 立即中止安装，提示完整性校验失败。 |
| 签名验证失败 | 中止安装，提示证书不可信。 |
| 证书已过期 | 警告用户，建议联系发布者或等待官方更新。 |
| 仓库地址不可达 | 降级为仅验证证书与哈希，提示网络隔离风险。 |
| 指纹在 index.json 中标记为 `retired` | 拒绝安装，提示该包已退役。 |

## 10. 安全与合规要求

### 10.1 哈希算法
- 当前强制使用 **SHA-256**（FIPS 180-4）。
- 若未来出现密码学突破，规范将升级至 SHA-3 或更高标准，并通过 `esso_version` 字段区分。

### 10.2 签名算法
- 当前强制使用 **Ed25519**（RFC 8032）。
- 私钥必须由发布者本地安全存储，官方不代管发布者私钥。

### 10.3 传输安全
- 所有服务器端点必须强制 **HTTPS**（TLS 1.3 或更高）。
- GitHub Pages 默认满足此要求。

### 10.4 审计与透明
- `index.json` 及所有证书文件均通过 Git 版本控制，任何变更留有提交记录。
- 鼓励第三方镜像定期爬取并校验官方服务器内容，形成分布式审计网络。

### 10.5 降级攻击防护
- 客户端必须拒绝安装版本号低于已安装版本的包（除非显式强制降级）。
- 证书中 `certified_at` 时间戳用于检测异常回溯签发。
